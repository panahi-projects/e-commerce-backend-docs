# i18n

Translations live in `src/i18n/en/*.json` and `src/i18n/fa/*.json`. Setup is `nestjs-i18n` configured in `src/i18n/i18n.module.ts`.

## Resolution order

1. `?lang=fa` query param.
2. `?l=fa` query param.
3. `X-Lang` header.
4. `Accept-Language` header.
5. Fallback to `en`.

## Adding a translation key

1. Pick the right namespace file (`auth.json`, `common.json`, `tenants.json`, or create `<plugin-key>.json` for plugin keys).
2. Add the key to **both** `en/<file>.json` and `fa/<file>.json`. Missing translations fall back to the key string, which is a code smell.
3. Reference it from code as `<file>.<key>`. Example: `throw new BadRequestException('tenants.not_found')`.

## Adding a language

1. Create `src/i18n/<lang>/` mirroring the `en/` directory.
2. Translate every key.
3. `nest-cli.json` already globs `i18n/**/*.json` into the build assets — no change needed.

## Where i18n keys are used

- `HttpExceptionFilter` does **not** automatically translate. The current convention is that controllers throw `HttpException`s with i18n keys as the message; the frontend client resolves them, or you can add an interceptor that translates server-side via the `I18nService`.
- For error messages with parameters: `throw new NotFoundException({ message: 'common.not_found', args: { resource: 'Product' } })` and have the frontend resolve `{{resource}}`.

## Plugin-aware translations

Plugins should ship their own translation files when they add user-facing messages. Naming convention: `src/i18n/<lang>/<plugin-key>.json` (camelCase plugin key in file names is OK but kebab-case is also used — `loyalty-points.json` and `compare-products.json` are precedented).
